- **Real edge case found during the forced read-only review, fixed before committing**: `isSpeechAvailable()` only ever checked whether the `SpeechRecognition` constructor exists on `window` — it can't detect whether recognition actually *works*. iOS Safari specifically has a long history of exposing the constructor while recognition silently fails (mic permission denied, no mic, or the platform's own speech service refusing) — without a fix, a child on an affected device would have seen the mic UI, tried to speak, and gotten `heard:''` → similarity 0 → "incorrect" on every single attempt forever, with zero path to the listening-mode fallback the spec explicitly asks for as the whole point of graceful degradation. Fixed by adding `isPermissionError(code)` (distinguishing a real permission/hardware/service failure from an ordinary "didn't catch that" — no-speech/aborted/network, which is just worth a plain retry) and threading the recognition's actual `error` code through `startListening`'s resolved result. `ReadAloud.jsx` now sets a `speechBroken` flag the moment a real permission-type error fires, which flips the UI to listening mode for the rest of that round (reset fresh at the start of each new round) rather than repeatedly penalizing the child for a systemic failure that isn't their fault.
- **A second, related timing bug caught in the same review**: the original 6-second listening timeout counted down from the moment `.start()` was called — but on a device's very first mic-permission request, the OS's native permission dialog itself can take several seconds to resolve, silently eating into the child's real speaking window before they'd even seen the prompt or had a chance to respond. Fixed by restarting the timeout from the real `onaudiostart` event (mic capture actually beginning) instead, falling back to counting from `.start()` if `onaudiostart` never fires (older/less-standard implementations).
- Verified the fix itself the same way as the original logic: added `isPermissionError()` test cases to the committed `speech.test.js` suite (distinguishing `not-allowed`/`service-not-allowed`/`audio-capture` from `no-speech`/`aborted`/`network`/`null`), and extended `SpeechTestHarness.jsx` to surface the raw `errorCode` + a "would fall back to listening mode here" flag in its on-screen log, so a real-device tester can directly observe whether this exact class of failure is what's actually happening on their phone, rather than just seeing an opaque "incorrect."
- Verification overall: `npm run build` clean; `npm test` 41/41 pass (12 new `speech.test.js` + 4 new `readAloudWords.test.js`, on top of the 29 pre-existing from earlier today's §1.1-§1.3 + cleanup work). The actual `SpeechRecognition` behavior, the `ReadAloud.jsx`/`SpeechTestHarness.jsx` UI, and — critically — whether the new `speechBroken` fallback genuinely engages against a REAL permission-denial dialog (as opposed to just the unit-tested classification logic) were never exercised live this session, for the same no-microphone/no-browser reason as every other UI piece built today. ## What already exists in Supabase (do not recreate)

### Tables

- `public.friend_codes` — `user_id` (PK, FK→auth.users), `code` (text, unique, 6 chars), `created_at`
- `public.friendships` — `id` (PK uuid), `requester_id`, `target_id`, `status` ('pending'|'accepted'|'rejected'), `created_at`, `responded_at`

### View

- `public.my_friends` — already filtered by RLS to the logged-in user. Columns: `friendship_id`, `friend_user_id`, `friend_name`, `created_at`, `responded_at`. Only contains `status = 'accepted'` rows.

### RPC functions (call via `supabase.rpc(...)`)

1. **`ensure_friend_code(p_user_id uuid)`** → returns `text` (the code)
   - Idempotent. Call once right after a user's first egg is created.
   - Generates a permanent 6-char code (charset excludes O/0, I/1, L — kid-safe, no ambiguous chars).

2. **`send_friend_request(p_code text)`** → returns table `(friendship_id uuid, status text)`
   - The ONLY way to look up another user — by exact code match.
   - Handles: code not found (raises exception), self-friend attempt (raises exception), existing pending/accepted relationship (returns existing row instead of duplicating).
   - Auth required (`auth.uid()` used internally as requester).

3. **`respond_friend_request(p_friendship_id uuid, p_accept boolean)`** → returns `text` (new status: 'accepted' or 'rejected')
   - Only callable by the `target_id` of a pending request.
   - Raises exception if not authorized or already resolved.

### Direct table reads (RLS-protected, safe to query directly)

- `select * from friend_codes where user_id = auth.uid()` → get my own code to display
- `select * from friendships where target_id = auth.uid() and status = 'pending'` → incoming requests to show "Accept/Reject" UI for
- `select * from friendships where requester_id = auth.uid() and status = 'pending'` → outgoing requests I'm waiting on (show "Pending..." state)
- `select * from my_friends` → my accepted friends list
