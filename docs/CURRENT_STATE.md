### Speaking & Reading-Aloud Mode (Learning Core Overhaul В§1.4 вҖ” 2026-07-09)
- **`src/lib/speech.js`** вҖ” thin wrapper over the browser's Web Speech API (`SpeechRecognition`/`webkitSpeechRecognition`). `isSpeechAvailable()` feature-detects (false in any non-browser context, e.g. this project's Node test runner). `startListening(expected, {lang, timeoutMs})` returns `{ promise, stop }` вҖ” `stop()` calls the recognition's own graceful `.stop()` (not `.abort()`) so a "hold to talk" release still resolves with whatever was actually heard rather than discarding it; `listenForPhrase()` is a simple fire-and-forget convenience wrapper for callers that don't need early-stop (e.g. the dev harness below). Falls back to `{ heard:'', match:0, unavailable:true }` immediately when unsupported.
  - **Similarity**: `normalize()` lowercases, strips punctuation, and strips ONLY Thai tone marks (а№„аёЎа№үа№ҖаёӯаёҒ/а№Ӯаё—/аё•аёЈаёө/аёҲаёұаё•аё§аёІ) вҖ” deliberately NOT other Thai diacritics/vowel marks, since those change the word's actual identity (аёӘаёЈаё° mismatches are real misses, not just accent variance a young reader's imperfect tone shouldn't be penalized for). `levenshteinDistance()` + `similarity()` (standard `1 - distance/maxLength` ratio, 0-1). `classifyMatch()`: вүҘ0.7 `'correct'`, 0.4-0.7 `'almost'`, <0.4 `'incorrect'` (spec thresholds, boundaries inclusive on the lower side of each tier).
  - **Privacy** (code comment + this note, per instruction): recognition runs entirely inside the browser's own built-in speech API вҖ” this file never records, stores, or transmits raw audio anywhere. It only ever reads the short text transcript the browser's API returns, compares it locally, and discards it. Nothing is uploaded to any KidQuest server.
  - **Real edge case found+fixed during this session's own read-only review** (done while a temporary tool-availability gap blocked running build/test вҖ” used the wait productively rather than idly): `isSpeechAvailable()` only detects whether the `SpeechRecognition` constructor *exists*, not whether recognition actually *works*. iOS Safari specifically is known to expose the constructor while recognition silently fails (permission denied, no mic, service refusal) вҖ” without a fix, that would have meant every attempt just resolves to `heard:''` вҶ’ similarity 0 вҶ’ "incorrect" forever, with the child stuck and no path to the listening-mode fallback the spec explicitly calls for. Added `isPermissionError(code)` to distinguish a real permission/hardware/service failure from an ordinary "didn't catch that" (no-speech/aborted/network вҖ” worth a plain retry, not a fallback), threaded `errorCode` through `startListening`'s resolved result. Also fixed a related timing bug: the `timeoutMs` window now restarts from the real `onaudiostart` event (mic capture actually beginning) instead of counting from `.start()` вҖ” a first-time OS permission dialog can itself take several seconds, and counting the timeout from `.start()` would silently eat into the child's real speaking window before they'd even seen the prompt.

### Persistence & Auth
- localStorage key `kq_state` вҖ” always-on, written every state change
- Supabase `eggs` table: `user_id, child_name, state_json, updated_at`
- **Mandatory login** вҖ” app fully blocked behind email/password auth; no guest mode
- **Onboarding gate** вҖ” new accounts must set name + schoolGrade + gender before app is accessible; `state.name === ''` is the trigger
- `state.schoolGrade` вҖ” parent-entered label string (e.g. "аёӣ.1"), informational only, does not affect game progression
- `state.gender` вҖ” `'male' | 'female' | 'unspecified'`
- `state.stateVersion` вҖ” schema version (currently 1); `migrateStateShape()` deep-merges new nested fields on load
- `state.lastSavedAt` вҖ” timestamp used for cloud conflict resolution (replaces `rounds` counter). **`defaultState()` now sets this to `0`, not `Date.now()`** (fixed 2026-07-03 вҖ” see below): `0` means "never actually saved," so a genuinely blank/new local state can never masquerade as a recent real save. Only `saveState()` stamps a real `Date.now()`.
- `resolveSync(local, remote)` вҖ” single source of truth for cloud conflict resolution in `state.js`. Whole-object last-write-wins by `lastSavedAt` (tie вҶ’ remote), but **before** that fallback, two additional safety nets run (added 2026-07-03, alongside the pre-existing `hasRemoteCreatures && !hasLocalCreatures` check): (1) `local.lastSavedAt === 0 && remote.lastSavedAt > 0` вҶ’ remote wins outright; (2) `hasRealProgress(remote) && !hasRealProgress(local)` вҶ’ remote wins вҖ” `hasRealProgress()` (new export in `state.js`) checks `hatchedEggs`/`xpThai+xpEng+xpMath`/`rounds`/`ownedItems`/`ownedRoomItems`/`grade`/`badges`, deliberately EXCLUDING `coins`/`happiness`/login fields, because those are the fields the first-mount maintenance dispatches (`DECAY_HAPPINESS`/`CHECK_DAILY_RESET`/`DAILY_LOGIN`/`ER_SAVE_SCORE`) can touch before the async cloud fetch resolves вҖ” using a maintenance-immune signal is what actually closes the cross-device data-loss race for a *returning* player (not just a literally-blank account), since those dispatches would otherwise inflate `local.lastSavedAt` before `resolveSync` ever runs. **Any reducer that mutates persisted state MUST stamp `lastSavedAt: Date.now()`**, otherwise a stale remote snapshot can win a later resolveSync and silently revert the change on reload (bugs fixed this way: `c74e83d` daily-login coins; 2026-07-01 cosmetics/room purchases). Reducers currently stamping it: `DAILY_LOGIN`, `BUY_ITEM`, `EQUIP_ITEM`, `BUY_ROOM_ITEM`, `PLACE_ROOM_ITEM`, `REMOVE_ROOM_ITEM`, and (per the 2026-07-01 full-sweep) all mutating reducers including every minigame `*_DEDUCT_LIFE` case.
- **Initial-sync gate** (`state.js`, 2026-07-03): a module-level `_initialSyncComplete` flag (`markInitialSyncComplete()`/`isInitialSyncComplete()`) blocks `saveState()` entirely вҖ” no localStorage write, no Supabase push вҖ” until the mount-time `loadState()` вҶ’ `resolveSync()` вҶ’ `dispatch(INIT)` chain has resolved (or safely failed; the chain now has a `.catch()` + `.finally(markInitialSyncComplete)` in `StateContext.jsx` so a network failure never permanently disables saving). This closes the actual race: without it, the very first render's `useEffect(() => saveState(state), [state])` could push an empty `defaultState()` to Supabase before the real cloud row had been fetched, overwriting real cross-device progress.
- **Blank-wipe guard** (`state.js`, 2026-07-03): both `saveState()` and `syncToSupabase()` independently refuse to persist/push a state whose `lastSavedAt` is falsy AND that carries no `hasRealProgress()` signal вҖ” belt-and-suspenders on top of the initial-sync gate, and the only guard that protects `syncToSupabase()`'s direct callers that bypass `saveState()` (the migration re-push inside `loadState()`, the `SIGNED_IN` auth listener).
- `SaveStatusIndicator.jsx` вҖ” fixed bottom-right badge showing saving/saved/error/offline via pub/sub (`onSaveStatusChange`)
- Logout: `supabase.auth.signOut()` + `localStorage.removeItem(KEY)` + `dispatch(INIT, defaultState())` вҖ” fully clears local state
- `_migrateBattleStats()` backfills all new fields for legacy eggs on load
- One-time migration flags: `_subjectLevelCalibrated` (recalibrate subjectLevels from levelMastery), `_itemsMigrated` (additive itemsвҶ’homeItems/battleItems merge), `_evoRechecked` (recheck all creature evoStage on load)
