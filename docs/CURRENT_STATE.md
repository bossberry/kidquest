### Speaking & Reading-Aloud Mode (Learning Core Overhaul ¬ß1.4 ‚Äî 2026-07-09)
- **`src/lib/speech.js`** ‚Äî thin wrapper over the browser's Web Speech API (`SpeechRecognition`/`webkitSpeechRecognition`). `isSpeechAvailable()` feature-detects (false in any non-browser context, e.g. this project's Node test runner). `startListening(expected, {lang, timeoutMs})` returns `{ promise, stop }` ‚Äî `stop()` calls the recognition's own graceful `.stop()` (not `.abort()`) so a "hold to talk" release still resolves with whatever was actually heard rather than discarding it; `listenForPhrase()` is a simple fire-and-forget convenience wrapper for callers that don't need early-stop (e.g. the dev harness below). Falls back to `{ heard:'', match:0, unavailable:true }` immediately when unsupported.
  - **Similarity**: `normalize()` lowercases, strips punctuation, and strips ONLY Thai tone marks (ýπÑý∏°ýπâýπÄý∏≠ý∏Å/ýπÇý∏ó/ý∏ïý∏£ý∏µ/ý∏àý∏±ý∏ïý∏ßý∏≤) ‚Äî deliberately NOT other Thai diacritics/vowel marks, since those change the word's actual identity (ý∏™ý∏£ý∏∞ mismatches are real misses, not just accent variance a young reader's imperfect tone shouldn't be penalized for). `levenshteinDistance()` + `similarity()` (standard `1 - distance/maxLength` ratio, 0-1). `classifyMatch()`: ‚â•0.7 `'correct'`, 0.4-0.7 `'almost'`, <0.4 `'incorrect'` (spec thresholds, boundaries inclusive on the lower side of each tier).
  - **Privacy** (code comment + this note, per instruction): recognition runs entirely inside the browser's own built-in speech API ‚Äî this file never records, stores, or transmits raw audio anywhere. It only ever reads the short text transcript the browser's API returns, compares it locally, and discards it. Nothing is uploaded to any KidQuest server.
  - **Real edge case found+fixed during this session's own read-only review** (done while a temporary tool-availability gap blocked running build/test ‚Äî used the wait productively rather than idly): `isSpeechAvailable()` only detects whether the `SpeechRecognition` constructor *exists*, not whether recognition actually *works*. iOS Safari specifically is known to expose the constructor while recognition silently fails (permission denied, no mic, service refusal) ‚Äî without a fix, that would have meant every attempt just resolves to `heard:''` ‚Üí similarity 0 ‚Üí "incorrect" forever, with the child stuck and no path to the listening-mode fallback the spec explicitly calls for. Added `isPermissionError(code)` to distinguish a real permission/hardware/service failure from an ordinary "didn't catch that" (no-speech/aborted/network ‚Äî worth a plain retry, not a fallback), threaded `errorCode` through `startListening`'s resolved result. Also fixed a related timing bug: the `timeoutMs` window now restarts from the real `onaudiostart` event (mic capture actually beginning) instead of counting from `.start()` ‚Äî a first-time OS permission dialog can itself take several seconds, and counting the timeout from `.start()` would silently eat into the child's real speaking window before they'd even seen the prompt.
  - Verified via a committed regression suite (`src/lib/__tests__/speech.test.js`, 12 tests) covering `isSpeechAvailable`/`normalize`/`levenshteinDistance`/`similarity`/`classifyMatch`/`isPermissionError` ‚Äî the actual `SpeechRecognition` wrapper itself can't be unit-tested (needs a real browser + microphone), which is exactly what the dev harness below is for.
- **`src/lib/readAloudWords.js`** ‚Äî word/sentence pools per curriculum node, pulled directly from `wordPools.js` (NOT from `questionBank.js`'s `generateQuestion()`, since quiz-question shapes ‚Äî sequence tiles, numpad answers, memory-card pairs ‚Äî don't map onto "here's a word, say it out loud"). Covers all 14 thai + 14 eng nodes with real content; nodes with no clean single-word/sentence source (reading-comprehension passages, spelling/grammar nodes) fall through to a generic subject fallback (`TH_COMMON_WORDS`/`ENG_SIGHT_1`). `pickReadAloudWords(subject, nodeId, count)` shuffles and wraps if the pool is smaller than `count`.
- **`src/games/minigames/ReadAloud.jsx`** ("ý∏≠ýπàý∏≤ý∏ôýπÉý∏´ýπâýπÑý∏Çýπàý∏üý∏±ý∏á") ‚Äî registered in the existing minigame pool (`MINIGAMES.readaloud` in `minigameLives.js`: unlock at creature battleLevel 4, 3 lives/day; `THEMES.readaloud` in `minigameUI.jsx`), same ready‚Üíplaying‚Üídone phase pattern as the other 5 minigames. Each round picks thai OR eng at random, reads the child's real `activeNodes[subject]`, pulls 5 words via `readAloudWords.js`. Mic path (`SpeechRecognition` available): one-time friendly mic-permission intro screen (voiced in Thai) ‚Üí per word, a big üé§ press-and-hold button ‚Üí egg reacts live (listening state shown, tier-based feedback text). Match ‚â•0.7 ‚Üí correct, coins/XP. 0.4-0.7 "almost" ‚Üí retry once. <0.4 ‚Üí TTS models the correct pronunciation, then one repeat attempt ‚Äî max 2 tries per word either way, never traps the child. Coins = 3 √ó words fully matched (0-15 per round). **Listening-mode fallback** (`isSpeechAvailable()` false, e.g. some iOS Safari versions ‚Äî OR a `speechBroken` flag set mid-game the moment a real `isPermissionError` fires, since recognition can't ever succeed for the rest of that round once permission is genuinely denied/unavailable, and the child shouldn't be scored "incorrect" for a systemic failure that isn't their fault; resets fresh at the start of each new round): skips the mic UI entirely ‚Äî TTS speaks the target word, child taps 1 of 3 picture choices (target + 2 distractors from the same node pool) instead of speaking.
- **`src/components/SpeechTestHarness.jsx`** ‚Äî dev-only real-device verification tool, opened via `?speechtest=1` (checked in `App.jsx` before any auth/state gate, so it needs no login). Lets a real tester pick Thai/English + a preset or custom expected phrase, tap to record, and see the heard transcript + similarity score + tier + `errorCode` (flagging permission/hardware issues specifically, with a note that the real app would fall back to listening mode there) logged on screen (newest first) ‚Äî since this environment has no microphone to verify recognition quality directly. Never linked from the normal child-facing UI.
- **State**: `readAloudLives: 3`, `lastReadAloudDate: ''` ‚Äî same daily-lives convention as the other 5 minigames (no special `hasRealProgress`/migration handling needed: none of the other 5 minigames' lives counters are protected there either, since they're daily-resetting counters, not permanent progress ‚Äî `readAloudLives` deliberately matches that existing precedent rather than introducing an inconsistent exception). New `READALOUD_DEDUCT_LIFE` reducer, same shape as `FISHING_DEDUCT_LIFE` etc.

The pure logic (`speech.js`'s similarity/classification/permission-error detection, `readAloudWords.js`'s pool coverage) has a committed regression suite. The `ReadAloud.jsx`/`SpeechTestHarness.jsx` UI and the real `SpeechRecognition` behavior (including whether the `speechBroken` mid-game fallback actually triggers correctly on a real device that denies mic permission) were NOT verified live this session (no microphone/browser access) ‚Äî that's exactly what the harness page exists for; needs a real-phone pass (Thai + English, ideally including one deliberate permission-denial to confirm the fallback) before this can be called production-confident.

### Persistence & Auth
- localStorage key `kq_state` ‚Äî always-on, written every state change
- Supabase `eggs` table: `user_id, child_name, state_json, updated_at`
- **Mandatory login** ‚Äî app fully blocked behind email/password auth; no guest mode
- **Onboarding gate** ‚Äî new accounts must set name + schoolGrade + gender before app is accessible; `state.name === ''` is the trigger
- `state.schoolGrade` ‚Äî parent-entered label string (e.g. "ý∏õ.1"), informational only, does not affect game progression
- `state.gender` ‚Äî `'male' | 'female' | 'unspecified'`
- `state.stateVersion` ‚Äî schema version (currently 1); `migrateStateShape()` deep-merges new nested fields on load
- `state.lastSavedAt` ‚Äî timestamp used for cloud conflict resolution (replaces `rounds` counter). **`defaultState()` now sets this to `0`, not `Date.now()`** (fixed 2026-07-03 ‚Äî see below): `0` means "never actually saved," so a genuinely blank/new local state can never masquerade as a recent real save. Only `saveState()` stamps a real `Date.now()`.
- `resolveSync(local, remote)` ‚Äî single source of truth for cloud conflict resolution in `state.js`. Whole-object last-write-wins by `lastSavedAt` (tie ‚Üí remote), but **before** that fallback, two additional safety nets run (added 2026-07-03, alongside the pre-existing `hasRemoteCreatures && !hasLocalCreatures` check): (1) `local.lastSavedAt === 0 && remote.lastSavedAt > 0` ‚Üí remote wins outright; (2) `hasRealProgress(remote) && !hasRealProgress(local)` ‚Üí remote wins ‚Äî `hasRealProgress()` (new export in `state.js`) checks `hatchedEggs`/`xpThai+xpEng+xpMath`/`rounds`/`ownedItems`/`ownedRoomItems`/`grade`/`badges`, deliberately EXCLUDING `coins`/`happiness`/login fields, because those are the fields the first-mount maintenance dispatches (`DECAY_HAPPINESS`/`CHECK_DAILY_RESET`/`DAILY_LOGIN`/`ER_SAVE_SCORE`) can touch before the async cloud fetch resolves ‚Äî using a maintenance-immune signal is what actually closes the cross-device data-loss race for a *returning* player (not just a literally-blank account), since those dispatches would otherwise inflate `local.lastSavedAt` before `resolveSync` ever runs. **Any reducer that mutates persisted state MUST stamp `lastSavedAt: Date.now()`**, otherwise a stale remote snapshot can win a later resolveSync and silently revert the change on reload (bugs fixed this way: `c74e83d` daily-login coins; 2026-07-01 cosmetics/room purchases). Reducers currently stamping it: `DAILY_LOGIN`, `BUY_ITEM`, `EQUIP_ITEM`, `BUY_ROOM_ITEM`, `PLACE_ROOM_ITEM`, `REMOVE_ROOM_ITEM`, and (per the 2026-07-01 full-sweep) all mutating reducers including every minigame `*_DEDUCT_LIFE` case.
- **Initial-sync gate** (`state.js`, 2026-07-03): a module-level `_initialSyncComplete` flag (`markInitialSyncComplete()`/`isInitialSyncComplete()`) blocks `saveState()` entirely ‚Äî no localStorage write, no Supabase push ‚Äî until the mount-time `loadState()` ‚Üí `resolveSync()` ‚Üí `dispatch(INIT)` chain has resolved (or safely failed; the chain now has a `.catch()` + `.finally(markInitialSyncComplete)` in `StateContext.jsx` so a network failure never permanently disables saving). This closes the actual race: without it, the very first render's `useEffect(() => saveState(state), [state])` could push an empty `defaultState()` to Supabase before the real cloud row had been fetched, overwriting real cross-device progress.
- **Blank-wipe guard** (`state.js`, 2026-07-03): both `saveState()` and `syncToSupabase()` independently refuse to persist/push a state whose `lastSavedAt` is falsy AND that carries no `hasRealProgress()` signal ‚Äî belt-and-suspenders on top of the initial-sync gate, and the only guard that protects `syncToSupabase()`'s direct callers that bypass `saveState()` (the migration re-push inside `loadState()`, the `SIGNED_IN` auth listener).
- `SaveStatusIndicator.jsx` ‚Äî fixed bottom-right badge showing saving/saved/error/offline via pub/sub (`onSaveStatusChange`)
- Logout: `supabase.auth.signOut()` + `localStorage.removeItem(KEY)` + `dispatch(INIT, defaultState())` ‚Äî fully clears local state
- `_migrateBattleStats()` backfills all new fields for legacy eggs on load
- One-time migration flags: `_subjectLevelCalibrated` (recalibrate subjectLevels from levelMastery), `_itemsMigrated` (additive items‚ÜíhomeItems/battleItems merge), `_evoRechecked` (recheck all creature evoStage on load)
